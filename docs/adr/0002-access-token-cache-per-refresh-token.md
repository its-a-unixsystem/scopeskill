# Access token cache keyed by refresh-token fingerprint

The `sv-cli` stores cached REST access tokens in per-identity files named by the first 16 hex characters of `sha256(REST_REFRESH_TOKEN)`, rather than a single shared cache file. A single file would cause cross-talk when the same machine runs against multiple scopeskill configs (e.g. agency dev with several customers): the access token minted for customer A would be served on a request authenticated against customer B's refresh token, producing confusing 401s until expiry. A keyed-entries-in-one-file design was rejected because it adds an in-file index and concurrent-write concerns the per-fingerprint scheme does not have. The cache directory is created `0700` and files `0600`, mirroring `.ssh` conventions, because the cached access token is a bearer credential.
